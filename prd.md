Documento de Requisitos do Produto (PRD)1. Visão Geral do ProdutoNome do Produto: Prancheta Tática DigitalObjetivo: Uma aplicação web voltada para analistas de desempenho, treinadores e criadores de conteúdo esportivo. O sistema permite a criação, animação e exportação de jogadas táticas de futebol, oferecendo ferramentas de desenho, simulação de movimentação em keyframes e gestão de jogadores no campo.2. Stack TecnológicoFrontend: HTML5 (utilizando a API nativa do <canvas> para renderização 2D dos elementos), Tailwind CSS (para UI, menus, painéis de controle e responsividade) e JavaScript (para a lógica de interação e animação).Backend: Node.js com Express (ou Python com FastAPI, à sua escolha) para gerenciamento das rotas de exportação e chamadas de banco de dados.Banco de Dados: SQLite (leve, local/baseado em arquivo, ideal para salvar os dados persistentes de times e jogadas).3. Requisitos Funcionais (Regras de Negócio)3.1. Gestão de Entidades e Camadas (O Canvas)Fundos Dinâmicos: O usuário deve poder alternar o background do canvas entre: campo inteiro, meio-campo (defensivo/ofensivo), campo reduzido e grande área.Posicionamento Relativo: Ao trocar o layout do campo, o sistema deve recalcular e manter a posição proporcional (coordenadas relativas $X, Y$) dos tokens já inseridos.Limites de Equipe: O sistema deve limitar as instâncias ativas a um máximo de 11 jogadores por equipe de linha.Categorização: Elementos devem ser divididos em três grupos visuais distintos: Equipe A (Mandante), Equipe B (Visitante) e Arbitragem, controlados por cores de uniforme personalizáveis.3.2. Mecânica de Movimentação e FísicaDrag and Drop: Todos os tokens (jogadores e bola) devem possuir a propriedade de serem arrastados e soltos livremente pelo plano 2D.Vínculo da Bola (Snap to Player): Se a bola for movida para as coordenadas (ou raio de colisão) de um jogador, ela adquire um estado de "posse". A bola passa a compartilhar as coordenadas de movimento desse jogador até que um evento de desvinculação (passe, chute ou perda de posse) seja disparado.3.3. Tinta Digital e Anotações TáticasAncoragem: As setas desenhadas devem ser tratadas como objetos vetoriais com pontos de origem e destino. A origem deve, opcionalmente, poder ser ancorada ao ID de um jogador, acompanhando seu movimento.Padrões de Traçado: O sistema deve ter três ferramentas específicas de desenho de rotas:Linha Contínua com Seta: Representa a movimentação de um jogador sem a posse da bola.Linha Tracejada com Seta: Representa a trajetória da bola solta (passe, lançamento, cruzamento).Linha Ondulada: Representa a condução de bola por um jogador.3.4. Motor de Animação (Linha do Tempo)Gravação por Keyframes: A aplicação deve possuir uma Timeline onde cada estado do campo (posição de cada token) é salvo como um Frame.Interpolação Matemática: Ao reproduzir a animação entre o Frame 1 e o Frame 2, o motor JavaScript do frontend calculará automaticamente a transição linear das coordenadas $X$ e $Y$ para criar uma movimentação fluida.3.5. Renderização e Modos de VisãoBase 2D: Toda a lógica matemática principal (distâncias, colisões, rotas) é resolvida em um plano cartesiano 2D ($X, Y$).Visualização 3D (Módulo Futuro/Conversor): O sistema deve ser projetado de forma modular. Caso uma engine 3D (como Three.js) seja acoplada no futuro, ela apenas consumirá as matrizes 2D já existentes e aplicará uma projeção isométrica ou de câmera em primeira pessoa para a visualização.3.6. Exportação e CompartilhamentoExportação Estática: Geração da imagem do canvas atual no formato .PNG ou .JPEG.Exportação Animada: Uma rotina de gravação do canvas (usando MediaRecorder API no frontend ou processamento via FFmpeg no backend) que compila os keyframes interpolados e gera um arquivo .MP4 pronto para upload (ex: YouTube).4. Estrutura do Banco de Dados (SQLite)O banco relacional precisará de uma estrutura simples para manter o estado das configurações e entidades base.Tabelas PrincipaisTabelaCampos (Colunas)DescriçãoTeamsid, name, primary_color, secondary_colorRegistra as equipes configuradas.Playersid, team_id, name, number, positionBanco simples de atletas vinculados a um time.Playsid, title, field_layout, created_atSalva o projeto/jogada (ex: "Saída de Bola 4-3-3").Framesid, play_id, frame_order, state_jsonO campo state_json guarda a matriz $X, Y$ de todos os tokens e linhas daquele quadro-chave.5. UI/UX e Frontend (Tailwind CSS)Barra Lateral (Esquerda): Biblioteca de times, arrastar jogadores para o campo, seleção do layout de fundo.Barra Superior: Ferramentas de anotação (tipos de linhas, cores de caneta, borracha).Área Central: O Canvas (o campo propriamente dito).Barra Inferior: Linha do tempo de animação, botão "Adicionar Frame", Play/Pause e botões de Exportação.
